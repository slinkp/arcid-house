# Drum Machine POC — RCade Cabinet

## Context

Browser-based drum sequencer targeting the RCade arcade cabinet.
Display resolution: **320×240px**. Input: joystick, two buttons.

This document describes the minimal proof-of-concept spike. Follow phases in order.
Do not add features outside scope without checking the Future Considerations section first.

---

## Tech Stack

- **Rcade** - we are building on this, see docs at https://github.com/fcjr/RCade/blob/main/README.md, we are starting from the code already bootstrapped via `npm create rcade@latest` 
- **Vanilla JavaScript** — Application code is in `src/main.js`. Initially this was generated by bootstrapping.
- single `index.html` in project root. Initially this was generated by bootstrapping.
- Single CSS file at `src/style.css`. Initially this was autogenerated by bootstrapping; anything unnecessary can be removed.
- **Tone.js** — audio engine, we need to bundle this via npm
- **`@rcade/plugin-input-classic`** — RCade input plugin for joystick and button access and keyboard testing; we have this already.
- **No other framework** — no React, no TypeScript, etc
- **Local testing**: `npm run dev`

## Architecture Constraints

### Design for RCade Sandbox

From the RCade docs, this is important:

```
    Bundle everything - Include all assets in your build. No external CDNs.
    No persistence - Games start fresh every time. Design for it.
    Use plugins for input - The @rcade/plugin-input-classic plugin handles all arcade controls.
    Keep it self-contained - Your game should work completely offline.

```

### Audio Engine Abstraction

All audio calls must go through a single `AudioEngine` object. Sequencer logic never calls Tone.js directly. This allows future swap to superdough or another engine.

```javascript
const AudioEngine = {
  init() {},
  setBPM(bpm) {},
  play() {},
  stop() {},
  triggerDrum(sampleName, time, velocity) {},
  loadSample(name, url) {},
  // stub — not implemented in POC:
  triggerNote(note, duration, time, params) {},
}
```

### Cursor Model

Use a 2D cursor from day one, even though only one row exists in this POC.

```javascript
const cursor = { x: 0, y: 1 }; // y:0 reserved for global controls row
```

### Pattern Data

Flat array of 0/1 values, 16 steps, hardcoded length.

```javascript
const pattern = [1,0,0,0, 1,0,0,0, 1,0,0,0, 1,0,0,0];
```

## Implementation Phases

### Phase 1 — Audio Playback

- [ ] Add Tone.js, bundled, we use npm
- [ ] Implement `AudioEngine` stub object
- [ ] use `Tone.MembraneSynth` for basic kick sound
- [ ] Implement `AudioEngine.triggerDrum()` wrapping the player/synth
- [ ] Define `pattern` array with quarter notes: `[1,0,0,0,1,0,0,0,1,0,0,0,1,0,0,0]`
- [ ] Wire up `Tone.Sequence` iterating 16 steps, calling `triggerDrum` on `1`s
- [ ] Hardcode BPM to 120
- [ ] **Auto-start playback on page load** — call `Tone.start()` and `Tone.Transport.start()` on first user gesture (click/keypress), then loop forever
- [ ] Verify kick plays in browser before proceeding

### Phase 2 — Step Sequencer UI

- [ ] Render 16 buttons in a horizontal row using CSS grid
- [ ] Each button visually reflects `pattern[i]`: lit (active) vs dark (inactive)
- [ ] Target button size: ~16px wide, fit within 320px with minimal padding
- [ ] Define CSS classes: `.step`, `.step-active`, `.step-cursor`, `.step-playing`
      These must be visually distinct from each other in all combinations (except that step and step-active are mutually exclusive).


### Phase 3 — Joystick Cursor Navigation

Start with this stub for input controls, and modify as needed:

```
import { PLAYER_1, PLAYER_2, SYSTEM } from "@rcade/plugin-input-classic";

function gameLoop() {
  // D-pad directions
  if (PLAYER_1.DPAD.up) moveUp();
  if (PLAYER_1.DPAD.down) moveDown();
  if (PLAYER_1.DPAD.left) moveLeft();
  if (PLAYER_1.DPAD.right) moveRight();

  // Action buttons
  if (PLAYER_1.A) fire();
  if (PLAYER_1.B) jump();

  // System buttons
  if (SYSTEM.ONE_PLAYER) startOnePlayerGame();
  if (SYSTEM.TWO_PLAYER) startTwoPlayerGame();

  requestAnimationFrame(gameLoop);
}
```

- [ ] Implement `cursor = { x: 0, y: 1 }`
- [ ] `PLAYER_1.DPAD.left` / `PLAYER_1.DPAD.right`: decrement/increment `cursor.x`, wrap at 0 and 15
- [ ] `PLAYER_1.DPAD.up` / `PLAYER_1.DPAD.down`: reserved for future row navigation, no-op for now
- [ ] Render cursor highlight on `buttons[cursor.x]` using `.step-cursor` class
- [ ] `PLAYER_1.A`: toggle `pattern[cursor.x] ^= 1`, update button class
- [ ] Do NOT add any computer keyboard navigation fallback: plugin-input-classic already does that

### Phase 4 — Playback Position Indicator

- [ ] In the `Tone.Sequence` callback, update a `playingStep` variable with current step index
- [ ] Trigger a lightweight UI update — re-apply classes to all 16 buttons each tick

### Phase 5 — Verification & Buffer

- [ ] Confirm loop plays forever without drift or stopping
- [ ] Confirm toggling steps while playing updates pattern immediately on next loop
- [ ] Confirm cursor and playing indicators are visually distinguishable at 320×240
- [ ] Basic smoke test: load page, hear kick, move cursor, toggle steps, observe changes

Use high contrast — white on black or similar. No subtle grays.

# Future Considerations

These are deferred but should **not** be designed against now.

**Row-based navigation** — The `cursor.y` field is already there. When adding global controls (start/stop, BPM) and a bass row, each row needs to know its own `maxX` so vertical movement can clamp `cursor.x` correctly. A simple row registry (`rows[y] = { length: n }`) is enough.

**Multiple drum tracks** — Adding more drum rows (snare, hihat, etc.) is just adding more `pattern` arrays and more UI rows. The sequencer loop should iterate over `tracks[]` rather than a single pattern before that point. Don't refactor now, but avoid making the single-pattern assumption load-bearing.

**AudioEngine swap** — Keep all Tone.js calls inside `AudioEngine`. We are very likely to switch to a better engine later. If you find yourself calling `Tone.*` outside that object, stop and move it. This is the one boundary worth enforcing strictly.

**Pattern length** — 16 steps is hardcoded. Don't parameterize it now, but don't store `16` as a magic number in more than one place — use a `const STEPS = 16`.

**More complex data** - we'll need more note info than ones and zeros eventually: pitch for sure.

**Sample loading** — Download and bundle audio samples for drum machine.

**TB303 bass synth** — Deferred entirely from this spike. When added, it gets its own UI row (`y=2`), its own pattern array, and routes through `AudioEngine.triggerNote()` which is already stubbed.
